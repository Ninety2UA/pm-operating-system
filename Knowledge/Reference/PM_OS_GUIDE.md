## 1. System Overview

### What is PM Operating System?

PM Operating System is an AI-powered personal productivity system built entirely on markdown files. It turns an AI assistant (Claude Code, ChatGPT, Cursor, or any MCP-compatible tool) into a personal task manager, project pipeline, and daily focus coach.

The core loop is simple:

1. **Brain dump** — Drop unstructured notes into `BACKLOG.md`
2. **Process** — Tell the AI to "process my backlog"
3. **Get organized** — The AI creates prioritized tasks and projects based on your goals
4. **Stay focused** — Ask "what should I work on?" and get smart suggestions

### Design Philosophy

- **Local-first**: All data lives on your machine as plain markdown. No cloud sync required.
- **Git-friendly**: Everything is markdown + YAML frontmatter. No binaries, no databases.
- **AI-native**: The system is designed for AI assistants to read, write, and reason about. The AI isn't bolted on — it's the primary interface.
- **Goal-driven**: Every task and project ties back to user-defined goals. The AI uses goals to prioritize work and flag misalignment.
- **Template-driven**: Consistent YAML frontmatter + markdown body across all files enables parsing, generation, and deduplication.
- **Extensible**: Skills, workflows, and integrations can be added without changing core structure.

### How It Works with AI Assistants

The AI assistant reads `AGENTS.md` (the system prompt / instruction file) which tells it:
- The workspace structure and what each directory contains
- How to process the backlog (classify, deduplicate, create files)
- Templates for tasks and projects
- The project lifecycle and when to decompose into tasks
- Daily guidance rules (max 3 focus items, flag blocked tasks, etc.)
- Specialized workflows to invoke for specific task types
- Skills to call for research, idea discovery, and validation

The AI never writes code — it stays within markdown and task management. It reads and writes markdown files, calls MCP tools for task operations, and invokes skills for web research.

## 3. Core Files

### CLAUDE.md

The entry point for Claude Code. Contains a single line:

```
@AGENTS.md
```

This tells Claude Code to read `AGENTS.md` for all system instructions. The indirection allows `AGENTS.md` to serve as the universal instruction file for any AI assistant, while `CLAUDE.md` is Claude Code-specific.

### AGENTS.md

**The brain of the system.** A ~200-line markdown file that serves as the complete instruction set for the AI assistant. It defines:

- The workspace structure and what goes where
- The difference between Tasks and Projects
- The backlog processing flow
- Task and Project templates (YAML frontmatter + markdown body)
- The project lifecycle (idea → evaluating → active → paused → archived)
- Goals alignment rules
- Daily guidance behavior (max 3 focus tasks, flag blocked items)
- Categories for task classification
- Specialized workflows and when to invoke them
- The list of MCP tools available
- Interaction style (direct, friendly, concise)

**Key rule embedded in AGENTS.md**: Every `prd.md` file MUST be generated by invoking the `/PRD` skill. Never write PRD content manually or through agents.

### GOALS.md

The user's strategic direction document. Generated by `setup.sh` from 5 questions. Structure:

```markdown
# Goals & Strategic Direction

*Last updated: [Date]*

## Current Context
### What's your current role?
### What's your primary professional vision?

## Success Criteria
### In 12 months, what would make you think 'this was a successful year'?
### What's your 5-year north star?

## Current Focus Areas
### What are you actively working on right now?
### What are your objectives for THIS QUARTER (next 90 days)?
### What skills do you need to develop?
### What key relationships or network do you need to build?

## Strategic Context
### What's currently blocking you or slowing you down?
### What opportunities are you exploring?

## Priority Framework
[P0-P3 definitions — see Section 5]

## What are your top 3 priorities right now?
```

The AI reads this file to:
- Prioritize tasks relative to quarterly objectives
- Suggest what to work on each day
- Flag tasks that don't align with any goal
- Ensure backlog items tie to strategic direction

**Update frequency**: Review and update weekly as priorities shift.

### BACKLOG.md

The raw capture inbox. No structure required — just dump text:

```markdown
# Backlog

Drop raw notes or todos here. Say `process my backlog` when you're ready for triage.

- need to follow up with james about the api thing
- idea: what if we added AI to the search?
- Q2 planning thoughts - mobile or web first?
- expense report overdue
```

During backlog processing, the AI reads this, classifies each item, creates tasks/projects, and clears the file.

**Important sections**: When the `/discover-ideas` skill runs, it appends structured findings under a `## Project Ideas` section with dated subsections.

## 4. File Formats & Templates

### Task File Format

**Location**: `Tasks/<task-name>.md`

```yaml
---
title: [Actionable task name]
category: [technical|outreach|research|writing|content|admin|personal|other]
priority: [P0|P1|P2|P3]
status: n  # n=not_started, s=started, b=blocked, d=done
created_date: YYYY-MM-DD
due_date: YYYY-MM-DD       # optional
estimated_time: [minutes]   # optional
resource_refs:
  - Knowledge/example.md    # optional — links to related Knowledge files
---

# [Task name]

## Context
Tie to goals from GOALS.md and reference any relevant material.

## Next Actions
- [ ] Step one
- [ ] Step two
- [ ] Step three

## Progress Log
- YYYY-MM-DD: Notes, blockers, decisions.
```

**Key conventions**:
- File names use kebab-case: `fix-file-upload-limit.md`
- Context section MUST reference the relevant goal from GOALS.md
- Next Actions uses markdown checkboxes for trackable steps
- Progress Log entries are dated and include blockers/decisions

### Project Idea File Format (idea.md)

**Location**: `Projects/<project-name>/idea.md`

```yaml
---
title: [Project name]
category: [technical|outreach|research|writing|content|admin|personal|other]
priority: [P0|P1|P2|P3]
project_status: idea  # idea | evaluating | active | paused | archived
created_date: YYYY-MM-DD
estimated_time: [minutes]   # rough total estimate, optional
resource_refs:
  - Knowledge/example.md    # optional
---

# [Project name]

## Context
Tie to goals and explain why this project matters. Reference specific goals from GOALS.md.

## Scope
High-level phases or workstreams. NOT broken into individual tasks yet — that happens
when the project moves to "active" status.

- Phase 1: [Description]
- Phase 2: [Description]
- Phase 3: [Description]

## Progress Log
- YYYY-MM-DD: Notes, decisions, status changes.
```

**Key conventions**:
- Folder names use kebab-case: `Projects/google-ads-budget-pacing-simulator/`
- Scope stays high-level — decomposition into tasks happens only when `project_status` becomes `active`
- The `resource_refs` field is updated by the `/validate-project` skill when research briefs are generated

### Project PRD File Format (prd.md)

**Location**: `Projects/<project-name>/prd.md`

PRDs are ALWAYS generated by the `/PRD` skill — never written manually. The skill produces a consistent structure:

```markdown
# [Project Title] - Product Requirements Document

## Introduction
[One-paragraph summary of what the product does]

## Goals
- Goal 1
- Goal 2
- Goal 3

## User Stories

### US-001: [Story Title]
**Description:** As a [user], I want to [action] so that [outcome].
**Acceptance Criteria:**
- [ ] Criterion 1
- [ ] Criterion 2
- [ ] Criterion 3

### US-002: [Story Title]
...

## Functional Requirements
- **FR-1:** [Requirement]
- **FR-2:** [Requirement]
...

## Non-Goals (Out of Scope)
- [Explicit exclusion 1]
- [Explicit exclusion 2]

## Technical Considerations
[Architecture decisions, library choices, performance notes]

## Success Metrics
- [Quantifiable outcome 1]
- [Quantifiable outcome 2]

## Open Questions
- [Unresolved decision 1]
- [Unresolved decision 2]
```

### Knowledge File Format

**Location**: `Knowledge/*.md` or `Knowledge/research/{projects|topics}/*.md`

General knowledge files have no strict format — they're reference material (specs, briefs, meeting notes, research).

Research briefs (generated by `/validate-project` and `/research-topic` skills) follow the shared template. See [Section 12: Research Brief System](#12-research-brief-system).

### Library Item Format

**Location**: `Library/<category>/<descriptive-name>/<main-file>.md`

```yaml
---
title: Descriptive name
description: One-line summary of what it does and when to use it
tools: [claude-code, cursor, chatgpt]
tags: [code-review, debugging, rails]
created: YYYY-MM-DD
---

[The actual prompt/skill/agent content — clean and ready to use]
```

Main file naming by category:

| Category | Main file name |
|----------|---------------|
| `prompts/` | `prompt.md` |
| `systems/` | `system.md` |
| `skills/` | `SKILL.md` |
| `agents/` | `agent.md` |
| `commands/` | `command.md` |

## 5. Status & Priority Systems

### Task Statuses

Tasks use single-character status codes in YAML frontmatter:

| Status | Code | Meaning | When to use |
|--------|------|---------|-------------|
| Not Started | `n` | Task exists but no work has begun | Default for new tasks |
| Started | `s` | Actively being worked on | When the user begins work |
| Blocked | `b` | Cannot proceed — waiting on something | External dependency, awaiting input |
| Done | `d` | Completed | Task is finished |

Status transitions:
```
n (not started) → s (started) → d (done)
                → b (blocked) → s (started) → d (done)
```

### Project Statuses

Projects use descriptive status values:

| Status | Meaning | What happens |
|--------|---------|--------------|
| `idea` | Captured, not yet evaluated | Just an idea in the pipeline |
| `evaluating` | Being scoped and assessed | Running `/validate-project`, researching feasibility |
| `active` | Committed to building | Decompose scope into discrete tasks in `Tasks/` |
| `paused` | On hold | Reason noted in Progress Log |
| `archived` | Abandoned or completed | No longer active |

Status transitions:
```
idea → evaluating → active → done/archived
                  → paused → active (resume)
                  → archived (abandon)
```

**Critical rule**: When a project moves to `active`, its Scope section MUST be decomposed into individual task files in `Tasks/`.

### Priority Levels

| Priority | Label | Timeframe | WIP Limit | Description |
|----------|-------|-----------|-----------|-------------|
| **P0** | Critical/Urgent | This week | Max 3 | Directly advances quarterly objectives, time-sensitive |
| **P1** | Important | This month | Max 5-7 | Builds key skills, advances strategy, high-value |
| **P2** | Normal | Scheduled | No limit | Supports broader objectives, operational work |
| **P3** | Low | Someday/maybe | No limit | Admin tasks, speculative, no clear advancement value |

The AI enforces WIP limits: if more than 3 P0 tasks exist, it flags the overload.

## 6. Categories

Eight standard categories for classifying tasks and projects:

| Category | Code | Description | Example tasks |
|----------|------|-------------|---------------|
| **Technical** | `technical` | Build, fix, configure | Fix bug, deploy app, set up CI |
| **Outreach** | `outreach` | Communicate, meet | Follow up with contact, attend meeting |
| **Research** | `research` | Learn, analyze | Competitive analysis, tech evaluation |
| **Writing** | `writing` | Draft, document | Write spec, documentation, internal docs |
| **Content** | `content` | Blog posts, social media, public writing | LinkedIn post, blog article, newsletter |
| **Admin** | `admin` | Operations, finance, logistics | Expense report, invoicing, scheduling |
| **Personal** | `personal` | Health, routines | Exercise plan, habit tracking |
| **Other** | `other` | Everything else | Catch-all for unclassified items |

Categories are adjustable per user. Define your own in `AGENTS.md` or `config.yaml`.

## 7. The Backlog Flow

The backlog is the entry point for all new work. Here is the complete end-to-end flow:

### Step 1: Capture

Drop anything into `BACKLOG.md`. No structure needed:

```markdown
- need to follow up with james about the api thing
- idea: what if we added AI to the search?
- Q2 planning thoughts - should we focus on mobile or web first?
- expense report overdue
```

### Step 2: Trigger Processing

Tell the AI:
```
Process my backlog
```

### Step 3: AI Reads Context

The AI reads:
1. `BACKLOG.md` — extracts every actionable item
2. `Knowledge/` — searches for matching keywords, project names, dates
3. `GOALS.md` — understands current priorities and objectives
4. Existing `Tasks/` and `Projects/` — to detect duplicates

### Step 4: Classification

Each item is classified:

| Classification | Criteria | Action |
|---------------|----------|--------|
| **Task** | Single outcome, < ~2 hours | Create file in `Tasks/` |
| **Project** | Multi-step, exploratory, > ~2 hours | Create `Projects/<name>/idea.md` + invoke `/PRD` skill for `prd.md` |
| **Needs Clarification** | Ambiguous, lacks context or priority | Stop and ask the user |
| **Potential Duplicate** | Similar to existing task/project (>60% match) | Flag and ask whether to merge |

### Step 5: Deduplication

The `process_backlog_with_dedup` MCP tool handles this automatically:
- **Title similarity**: Fuzzy matching using SequenceMatcher (threshold: 0.6)
- **Keyword overlap**: Extracts keywords from titles and compares
- **Ambiguity detection**: Flags items with vague patterns ("think about", "maybe", "look into")
- **Category guessing**: Suggests categories based on keyword matching

### Step 6: Create Files

For each clear item:
- **Tasks**: Created with full YAML frontmatter + Context + Next Actions + Progress Log
- **Projects**: Create `idea.md` with Project Template, then invoke `/PRD` skill to generate `prd.md`

### Step 7: Present Summary

The AI shows a summary table:

```
Ready to create (5 items):
| Item | Category | Priority |
|------|----------|----------|
| Follow up with James | outreach | P1 |
| Submit expense report | admin | P1 |
...

Needs clarification (2 items):
1. "Q2 planning" — Is this research or a decision to make?
2. "Bug: file upload" — Is this for you or the bug tracker?

Potential duplicate (1 item):
- "AI search idea" — 68% similar to "Explore AI features for Q2"
```

### Step 8: Clear Backlog

After the user confirms, `BACKLOG.md` is cleared (or the processed section is removed).

## 8. Project Lifecycle

### The Full Pipeline

```
BACKLOG.md (raw idea captured)
    ↓
Projects/<name>/idea.md (project_status: idea)
    +
Projects/<name>/prd.md (generated by /PRD skill)
    ↓
/validate-project (optional: market research)
    ↓
Projects/<name>/idea.md (project_status: evaluating)
    ↓
"Activate <project>" (user decision)
    ↓
Projects/<name>/idea.md (project_status: active)
    +
Decompose scope into Tasks/ files
    ↓
Work on tasks → mark done
    ↓
Projects/<name>/idea.md (project_status: archived)
```

### Status Descriptions

- **idea**: Captured but not evaluated. Created during backlog processing. Has both `idea.md` and `prd.md` from the start.
- **evaluating**: Being actively assessed. Typically involves running `/validate-project` to get market research, reviewing the PRD, deciding feasibility.
- **active**: Committed to building. **MUST decompose Scope into discrete task files in `Tasks/`**. Each task references the project in its Context section.
- **paused**: On hold. Reason MUST be recorded in Progress Log. Can be resumed later.
- **archived**: Completed or abandoned. Final note in Progress Log.

### Activation = Decomposition

When a user says "Activate [project name]":

1. Read the project's `idea.md` Scope section
2. Read the project's `prd.md` for detailed requirements
3. Break the scope into discrete tasks (each < 2 hours)
4. Create a task file in `Tasks/` for each, with:
   - Context referencing the project
   - Appropriate priority and category
   - Clear Next Actions from the PRD
5. Update `idea.md` frontmatter: `project_status: active`
6. Log the activation in Progress Log

### PRD Requirement

**Every project gets a PRD.** The mandatory rule:

> Every `prd.md` file MUST be generated by invoking the `/PRD` skill (via the Skill tool). Never write PRD content manually or through agents.

This applies to:
- New projects created during backlog processing
- Existing projects that lack a PRD
- PRD refreshes or rewrites requested by the user

## 9. Goals Alignment

### How Goals Drive the System

Goals are the north star for all prioritization decisions:

1. **During backlog processing**: Every new task/project Context section MUST reference the relevant goal from `GOALS.md`
2. **During daily planning**: The AI selects focus tasks based on goal alignment + priority
3. **During weekly review**: Progress is measured against quarterly objectives
4. **When no goal fits**: The AI asks whether to create a new goal entry or clarify why the work matters
5. **Misalignment detection**: The AI flags active tasks that don't support any current goal

### GOALS.md Structure

Generated by `setup.sh` from 5 questions:

| Question | Section | Purpose |
|----------|---------|---------|
| What's your current role? | Current Context | Helps AI tailor advice |
| What's your primary professional vision? | Current Context | Long-term direction |
| 12-month success criteria? | Success Criteria | Annual goals |
| Quarterly objectives? | Current Focus Areas | Near-term targets |
| Top 3 priorities? | Top Priorities | Immediate focus |

The **Priority Framework** section (P0-P3 definitions) is also embedded in GOALS.md, providing the AI with the rules for assigning priority levels.

## 10. Workflows

Workflows are step-by-step guides stored in `examples/workflows/`. The AI reads the relevant workflow file when triggered and follows its instructions.

### 10.1 Morning Standup

**File**: `examples/workflows/morning-standup.md`
**Duration**: 2 minutes
**Trigger**: "What should I work on today?"

**What the AI does**:
1. Reads all tasks with status `n` (not started) or `s` (started)
2. Checks priority levels (P0 first, then P1)
3. Checks for any `active` projects with no corresponding tasks (flags for decomposition)
4. Suggests top 3 focus items with time estimates and goal alignment
5. Flags any blocked tasks and proposes next steps

**Variations**:
- "I'm overwhelmed. What's the ONE thing I should focus on?"
- "I only have 2 hours. What can I realistically finish?"
- "Remind me what I was working on yesterday."

**Tips**: Do this first thing, before email/Slack. Keep it under 2 minutes.

### 10.2 Backlog Processing

**File**: `examples/workflows/backlog-processing.md`
**Duration**: 5-10 minutes
**Trigger**: "Process my backlog", "Clear my backlog"

See [Section 7: The Backlog Flow](#7-the-backlog-flow) for the complete flow.

**When to use**:
- End of day: Process notes captured throughout the day
- Weekly: Clear accumulated ideas
- After meetings: Turn action items into tasks

**Key rules**:
- Tasks (single outcome, < 2 hrs) go to `Tasks/`
- Projects (multi-step, exploratory) get `idea.md` + `prd.md` in `Projects/`
- Ambiguous items get flagged for clarification
- Duplicates are detected and flagged

### 10.3 Weekly Review

**File**: `examples/workflows/weekly-review.md`
**Duration**: 15-30 minutes (5-minute quick version available)
**Trigger**: "What did I accomplish this week?"

**Four steps**:

1. **Review Completed Work**: "What did I accomplish this week?"
   - Tasks completed, grouped by goal alignment
   - Highlights and milestones
   - Time invested

2. **Check Goal Progress**: "How am I tracking against my quarterly goals?"
   - Status table for each quarterly goal (on track / needs attention / at risk)
   - Recommendations for rebalancing

3. **Identify Blockers**: "What's blocked or stalled?"
   - Blocked tasks with wait times and suggested actions
   - Stalled tasks (started but no progress) with "still a priority?" prompt

4. **Plan Next Week**: "Help me plan next week"
   - Must-do items (P0/P1)
   - Should-do items (P2)
   - If-time-allows items
   - Time budget (deep work / meetings / buffer)

**When to do it**: Friday afternoon, Sunday evening, or Monday morning.

### 10.4 Content Generation

**File**: `examples/workflows/content-generation.md`
**Trigger**: Any task with category `content`, `writing`, `outreach`, or `marketing`

**Steps**:

1. **Check voice samples**: Look for `Knowledge/voice-samples/` for the user's writing samples
2. **Check voice guide**: Look for `Knowledge/voice-guide.md` for extracted patterns
3. **Gather context**: Read relevant Knowledge files based on content type
4. **Draft content**: Apply voice principles (conversational, specific, no cliches)

**Voice principles** (defaults, overridden by user's voice guide):

**Creating a voice guide**: Collect 5-10 writing samples → save to `Knowledge/voice-samples/` → analyze patterns → create `Knowledge/voice-guide.md` with sections for Sentence Style, Openers, Signature Phrases, Never Say, Tone by Context.

## 13. MCP Server

### Overview

**File**: `core/mcp/server.py` (~1037 lines Python)
**Dependencies**: `mcp>=1.0.0`, `pyyaml>=6.0`, `anthropic>=0.18.0`

The MCP server provides structured tools that AI assistants can call for task management operations. It's optional — the system works without it (the AI can read/write markdown directly), but the server adds deduplication, validation, and statistics.

### Starting the Server

```bash
pip install pyyaml mcp
python core/mcp/server.py
```

Or configure in `.mcp.json` for automatic startup.

### Available Tools

#### Task Management

| Tool | Description | Parameters |
|------|-------------|------------|
| `list_tasks` | Filter and view tasks | `category`, `priority`, `status` |
| `create_task` | Create new task with metadata | `title`, `category`, `priority`, `estimated_time` |
| `update_task_status` | Change task status | `task_name`, `new_status` (n/s/b/d) |
| `get_task_summary` | Statistics and time estimates | None |
| `check_priority_limits` | Alert if P0/P1 overloaded | None |
| `get_system_status` | Full system overview | None |

#### Backlog Processing

| Tool | Description | Parameters |
|------|-------------|------------|
| `process_backlog` | Parse items from BACKLOG.md | None |
| `process_backlog_with_dedup` | Smart processing with duplicate detection | None |
| `clear_backlog` | Reset BACKLOG.md after processing | None |

#### Session Evaluations

| Tool | Description | Parameters |
|------|-------------|------------|
| `list_evals` | Browse past session evaluations | None |
| `generate_eval` | Create eval from session trace | `session_data` |
| `annotate_eval` | Add judgement/notes to eval | `eval_id`, `notes` |
| `get_eval_summary` | Statistics on eval success rates | None |

#### Maintenance

| Tool | Description | Parameters |
|------|-------------|------------|
| `prune_completed_tasks` | Auto-cleanup old done tasks | `days_old` (default: 30) |

### Deduplication System

The `process_backlog_with_dedup` tool includes:

- **Similarity detection**: Uses `difflib.SequenceMatcher` with configurable threshold (default: 0.6)
- **Keyword extraction**: Pulls keywords from titles for overlap scoring
- **Ambiguity detection**: Flags vague items with patterns like "think about", "maybe", "look into"
- **Category guessing**: Suggests categories based on keyword matching against category definitions
- **Clarification generation**: Automatically creates questions for ambiguous items

### Configuration

```yaml
# config.yaml (or DEDUP_CONFIG in server.py)
deduplication:
  similarity_threshold: 0.6    # How similar before flagging (0-1)
  check_keywords: true          # Enable keyword overlap checking

priority_limits:
  P0: 3                         # Max concurrent P0 tasks
  P1: 5                         # Max concurrent P1 tasks (soft limit)
```

### Task Content Generation

The MCP server auto-generates rich task bodies based on category:

| Category | Auto-generated content |
|----------|----------------------|
| `outreach` | Communication template with recipient, follow-up timing |
| `writing` | Outline structure, audience, key points |
| `technical` | Problem statement, approach, testing steps |
| `research` | Research questions, sources to check, deliverable format |
| `marketing` | Campaign brief, audience, channels, metrics |

## 16. Setup Process

The script includes a special block at the top for AI agents:

```yaml
  similarity_threshold: 0.6    # 0-1, higher = stricter matching
  check_keywords: true          # Enable keyword overlap detection

priority_limits:
  P0: 3                         # Max concurrent P0 tasks
  P1: 5                         # Max concurrent P1 tasks (soft limit)
```

### MCP Server Configuration

The MCP server reads configuration from:
1. Environment variable `MANAGER_AI_BASE_DIR` (base directory for Tasks/, BACKLOG.md, etc.)
2. Hardcoded defaults in `server.py` (dedup thresholds, directory paths)

## 22. Interaction Style & Agent Behavior

### How the AI Should Behave

These rules are defined in `AGENTS.md` and govern all AI interactions:

1. **Be direct, friendly, and concise** — No filler, no over-explaining
2. **Batch follow-up questions** — Ask all clarification questions at once, not one at a time
3. **Offer best-guess suggestions with confirmation** — Don't stall waiting for perfect info; propose and confirm
4. **Never delete or rewrite user notes outside the defined flow** — Respect the user's content
5. **Never write code** — Stay within markdown and task management
6. **Max 3 focus tasks** — Don't overwhelm with suggestions (unless user insists)
7. **Flag blocked tasks proactively** — Propose next steps or follow-up questions
8. **Always tie to goals** — Every task/project should reference GOALS.md
9. **Ask when unsure** — If an item lacks context, priority, or clear next step, ask before creating
10. **Use the defined tools** — Call MCP tools for task operations instead of manually editing files

### What the AI Should NOT Do

- Write code or create software
- Delete user files without the defined flow
- Create tasks for items that lack context (ask first)
- Suggest more than 3 focus items (unless asked)
- Write PRDs manually (must use `/PRD` skill)
- Ignore goal alignment
- Make large assumptions about user intent without asking

### The AI's Mental Model

The AI should think of itself as a personal chief of staff:
- Knows the user's goals and priorities
- Keeps the backlog clean and organized
- Surfaces the right work at the right time
- Flags risks and blockers proactively
- Connects daily tasks to quarterly objectives
- Handles the organizational overhead so the user can focus on actual work
